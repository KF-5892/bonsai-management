# 多角的レビュー 改善提案まとめ

**対応案の整理**: [レビュー対応案.md](./レビュー対応案.md)（各項目の推奨対応・候補を記載）

技術要件・データ定義・UIモックアップに対して、5つの専門的観点からレビューした結果の改善提案です。

---

## 1. UI/UXデザイン

### 1.1 モバイルファースト対応 【優先度：高】

年間スケジュール表（`min-width: 1100px`）はスマートフォンで横スクロール必須となる。盆栽の世話は屋外でスマホを片手に行うことが多いため、モバイル用の代替UIが必要。

- モバイル向け：縦型カード表示 or 月別リスト表示への切替
- レスポンシブブレークポイントの設計方針を明文化

### 1.2 ページ遷移・ナビゲーション設計 【優先度：高】

現在のモックはページ内アンカーリンク（一覧表/カード/スケジュール/お役立ち）のみ。盆栽が増えた際の画面遷移フローが未定義。

- 盆栽詳細 → スケジュール → 作業ログ間のナビゲーション設計
- パンくずリストや戻り導線の検討

### 1.3 盆栽詳細画面のモックアップ作成 【優先度：高】

`ui-sample-detail.png` は低解像度で内容が読めない。一覧系の画面はHTMLモックがあるが、詳細画面は未作成。

- 盆栽詳細画面のHTMLモックアップを追加
- 作業ログ入力フォーム、スケジュール編集画面のモックも検討

### 1.4 アクセシビリティ 【優先度：中】

状態バッジ（良好/要注意/不調）が色のみで区別されている。

- 色覚特性に配慮し、アイコンやパターンを併用
- WCAG 2.1 AA のコントラスト比を確認

### 1.5 空状態・オンボーディング 【優先度：中】

新規ユーザーが盆栽0本のときの画面が未設計。

- 初回利用時のガイド表示（最初の盆栽を登録しましょう等）
- 各画面のデータなし状態のデザイン

## 2. マーケティング・事業戦略

### 2.1 差別化ポイントの明確化 【優先度：高】

既存の植物管理アプリ（Planta、GreenSnap等）との違いが「盆栽特化」だけでは弱い。

- 年間スケジュール表のような専門家向け機能をキラーフィーチャーとして前面に押し出す
- ターゲットユーザー（初心者/中級者/専門家）のペルソナを明確化
- 競合アプリとの機能比較表を作成

### 2.2 ソーシャル機能の将来検討 【優先度：低】

盆栽愛好家はコミュニティ意識が強い（展示会、即売会、盆栽園巡り）。

- 写真共有・タイムライン機能の将来検討
- 展示会情報の共有機能

### 2.3 スケーラビリティ想定の見直し 【優先度：中】

「数百ユーザー規模」は控えめな想定。盆栽人口は日本だけで数十万人、海外にも大きなコミュニティがある。

- ターゲット規模の再定義（プロ向けに絞るか、広く狙うか）
- それに合わせたインフラ設計の調整

### 2.4 多言語対応の判断 【優先度：低】

海外の Bonsai コミュニティ（英語圏・欧州）は大きい。i18n を後回しにするなら判断理由を明確化。

- 開発前検討事項「7.1 言語」の決定を早期に行う

## 3. データ設計

### 3.1 care_logs の画像複数対応 【優先度：高】

現在 `care_logs.image_url` は単一カラム。1回の作業で複数写真を撮ることは一般的。

- 案A: `bonsai_media` に `log_id` FK を追加し、ログとメディアを関連付け
- 案B: `image_urls` を JSONField に変更
- 推奨は案A（既存のメディア管理と統合できる）

### 3.2 soft delete の統一 【優先度：中】

現在の方式が不統一。

| テーブル | 現在の方式 |
|----------|-----------|
| users | `status = 'deleted'`（文字列） |
| bonsai_plants | `archived`（真偽値） |

- 全テーブルで統一的な論理削除方針を策定
- `archived` と `deleted` は意味が異なる（アーカイブ＝参照可能、削除＝非表示）ため、使い分けルールを明文化

### 3.3 時系列データの成長対策 【優先度：中】

care_logs は盆栽1本 × 毎日の水やり = 年間365レコード。40本なら年14,600件、数年で数万件に達する。

- 古いログのアーカイブ or パーティション方針を検討
- 一覧表示時のページネーション・日付範囲フィルタを前提に設計

### 3.4 main_image_url と bonsai_media の同期 【優先度：中】

検証メモにも記載があるが、同期の具体的な仕組みが未定。

- アプリケーション層で同期（メディア登録・削除時に main_image_url を更新）
- または main_image_url を廃止し、bonsai_media の最新レコードから動的に取得

### 3.5 検索インデックス戦略 【優先度：低】

記事検索に `SearchVector` / `pg_trgm` の記載はあるが、care_logs のメモ検索も高頻度で使われるはず。

- 全文検索対象テーブルの一覧化
- インデックス戦略の統一方針を策定

## 4. フロントエンド実装

### 4.1 インタラクション方針の決定 【優先度：高】

フィルタ、ソート、折りたたみ、ページネーションの挙動がすべて静的モック。Django テンプレートでのインタラクション実現方法が未定義。

- 候補：HTMX / Alpine.js / vanilla JS
- 推奨：HTMX（Django テンプレートとの親和性が高い）+ Alpine.js（軽量な状態管理）
- 技術要件書に追記

### 4.2 Tailwind CSS のビルド方針 【優先度：高】

モックでは `cdn.tailwindcss.com` を使用しているが、本番では非推奨。

- `@tailwindcss/cli` or PostCSS でのビルドパイプラインを技術要件書に追記
- 未使用CSSのパージ設定

### 4.3 画像処理パイプライン 【優先度：高】

盆栽の写真は重要機能だが、アップロードからCDN配信までのフローが技術要件書に薄い。

- アップロード → リサイズ → サムネイル生成 → CDN配信のフロー設計
- 候補：`django-imagekit`、S3 + CloudFront、Cloudflare R2 + Images
- 遅延読み込み（`loading="lazy"`）の標準適用

### 4.4 ガントチャートの実装方針 【優先度：中】

CSS Grid ベースの静的実装だが、実データでは月跨ぎや動的な期間計算が必要。

- ライブラリ採用（Frappe Gantt 等）か自前実装かの判断
- データ量が増えた際のレンダリングパフォーマンス検証

### 4.5 オフライン対応の優先度 【優先度：低】

庭先で作業記録をつけるシーンでネット接続が不安定な場合がある。

- PWA / Service Worker の対応フェーズを明確化
- MVP では対応せず、Phase 2 以降で検討が妥当

## 5. 盆栽専門家のニーズ

### 5.1 作業種別の拡充 【優先度：高】

現在の `task_type`（潅水・施肥・剪定・植え替え・観察）では盆栽特有の作業をカバーできない。

追加すべき作業種別:

| 作業 | 説明 |
|------|------|
| 芽摘み | 春先の新芽管理 |
| 葉刈り | 落葉樹の葉を刈り取って小葉化を促す |
| 葉水 | 霧吹きでの葉への散水 |
| 針金かけ / 針金外し | 樹形作りの重要作業 |
| 消毒 | 殺虫剤・殺菌剤の散布 |
| ジン・シャリ作り | 枯れ枝・幹の加工 |
| 取り木 / 挿し木 | 繁殖作業 |

- `task_type` は Choices 固定ではなく、ユーザーによるカスタム追加を許可するか検討
- 少なくとも初期 Choices を上記レベルまで拡充

### 5.2 天候・気温との連動 【優先度：高】

盆栽管理で最も重要な判断要素は「今日の天気と気温」。水やり頻度は気温30度と15度で全く異なる。

- 天気API（OpenWeatherMap等）との連携を将来機能として検討
- 最低限、care_logs に「天候」「気温」のメモ欄を追加

### 5.3 写真の時系列比較 【優先度：中】

盆栽の成長・変化を追うには「去年の同時期の写真」との比較が極めて重要。

- ビフォーアフター表示（スライダー or 並列表示）
- 年次タイムライン（同月の写真を年ごとに並べる）
- bonsai_media にデータはあるため、UI/ビュー層の設計が必要

### 5.4 植え替え専用サマリー 【優先度：中】

「再来年植え替え」の判断は過去の植え替え実績に基づく。care_logs に埋もれさせず、専用ビューが欲しい。

- 最終植え替え日、次回予定、使用した用土配合の一覧表示
- 植え替え時期が近い盆栽のアラート表示

### 5.5 展示会管理 【優先度：低】

盆栽専門家にとって展示会は重要イベント。

- 展示会スケジュール管理
- 出品盆栽の選定・仕上げ作業の逆算スケジュール
- MVP後の将来機能として検討
