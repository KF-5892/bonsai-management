## 0. サマリ

- 不足が見つかったのは主に **盆栽個体（植え替え・基本属性）** と **お役立ち記事（カテゴリ）**。

## 1. 🔴 高優先：構造的に不足・複数画面で要求

### 1-1. 植え替え（鉢・用土）データ

- 根拠モック: `_5`（植え替え履歴）, `_11`（盆栽詳細）
- 現状: 植え替えは `logs.CareLog(task_type="repotting")` + 自由メモ `notes` のみ。
  モックは**鉢・用土・次回目安を構造化データ**として表示している。

| 項目名 | 英名(案) | 型イメージ | 対象モデル | 必須 | 根拠 | 説明 |
|---|---|---|---|---|---|---|
| 現在の鉢 | pot_info | 文字列 | `bonsai.BonsaiPlant` | △ | _11 | 例: `丸鉢 30cm`。種類・サイズの自由入力 |
| 現在の用土配合 | soil_mix | 文字列 | `bonsai.BonsaiPlant` | △ | _11 | 例: `赤玉土7:桐生砂3` |
| 次回植え替え目安 | next_repot_due | 日付/文字列 | `bonsai.BonsaiPlant` | △ | _5 | 例: `2025年春`。「次回植え替え時期です」アラートの判定元 |
| 植え替え時の鉢 | pot_info | 文字列 | （植え替え履歴） | △ | _5 | 履歴 1 件ごとの鉢 |
| 植え替え時の用土配合 | soil_mix | 文字列 | （植え替え履歴） | △ | _5 | 履歴 1 件ごとの用土 |
| 根の所見 | root_note | 文字列 | （植え替え履歴） | △ | _5 | 例: `根がかなり回っていた` |

- **対応方針（2案）**:
  - **A案**: `CareLog` に `pot_info` / `soil_mix` / `root_note` を追加し、`task_type=repotting` の時だけ使う。
    既存「3ソース合成」設計を崩さず最小変更。`BonsaiPlant` には「現在値」`pot_info`/`soil_mix`/`next_repot_due` を追加。
  - **B案**: 専用 `RepottingRecord` モデルを新設（bonsai FK + 鉢/用土/根所見/実施日 + 任意で CareLog FK）。
    植え替え履歴画面（_5）の表現力は高いが、テーブル追加でやや重い。
  - 推奨: MVP は **A案**。履歴は CareLog の repotting 抽出で `_5` を構成。

## 4. 充足確認済み（追加不要）

- **肥料マスタ** `logs.Fertilizer`: 肥料名 / 固形・液体（`form_type`）/ N-P-K（`n`,`p`,`k`）/ 有機（`is_organic`）/ メモ → モック `_8` 充足。
- **作業ログ** `logs.CareLog`: 対象盆栽 / 作業種別 / 日時 / 気温 / 天候 / メモ / 写真 / 肥料 / 肥料の量 / 状態評価(1〜5) → モック `_9`,`5_x` ほぼ充足（不足は 2-1 強風のみ）。
- **状態評価 5 段階** `logs.HealthEvaluation`: 最高/良好/普通/要注意/不調 → `5_x` 充足。
- **品種マスタ** `bonsai.BonsaiSpecies`: 和名/学名/カテゴリ/解説/月別作業カレンダー → `_1`,`_11` 充足。
- **月次アドバイス** `schedules.MonthlyAdvice`: 月/タイトル/本文/カテゴリ → `_1`,`_6` ほぼ充足（不足は 2-2 気候）。
