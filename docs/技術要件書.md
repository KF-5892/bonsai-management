# 盆栽管理アプリ 技術要件書

## 1. 機能要件（データ定義からの導出）

[データ項目定義調査結果.md](../データ項目定義調査結果.md) から、以下の機能領域を特定。

| 領域 | 主な要件 |
|------|----------|
| 認証 | メール+パスワード、Google OAuth、Apple Sign In |
| 権限 | 管理者（品種・記事の編集）、一般ユーザー（参照のみ） |
| 盆栽管理 | CRUD、品種マスタ参照、タグ、画像 |
| スケジュール | 繰り返し（日/週/月/年）、次回予定計算、通知 |
| 作業ログ | スケジュール連動 or アドホック記録、写真添付 |
| 通知 | 時間帯制限、オフセット指定、Web Push or メール |
| 記事 | Markdown本文、品種関連、公開/下書き |

## 2. 推奨技術スタック（Django・PaaS向け）

### 2.1 技術一覧

| レイヤー | 技術 | 選定理由 |
|----------|------|----------|
| **バックエンド** | Django 5.x | 成熟したフレームワーク、Admin でメタデータ管理が容易 |
| **API** | Django REST Framework (DRF) | REST API、認証・権限の標準サポート |
| **言語** | Python 3.12+ | Django の標準 |
| **DB** | PostgreSQL | トランザクション、JSONB、全文検索に対応 |
| **認証** | django-allauth | メール+パスワード、Google、Apple OAuth |
| **画像ストレージ** | django-storages + S3 互換 | 本番は S3 / Cloudflare R2、開発はローカル |
| **通知** | django-celery-beat + Celery | スケジュール実行、メール送信（Resend 等） |
| **UI** | Django テンプレート + Tailwind CSS | サーバーサイドレンダリング、軽量 |
| **デプロイ** | Railway / Render / Fly.io | Django 対応 PaaS |

### 2.2 認証・権限フロー

- **django-allauth** で `email` / `google` / `apple` をサポート
- **Django User** を拡張し、`UserProfile` で `display_name`、`time_zone`、`role` 等を保持
- **role**: `admin` / `user`。管理者のみ `bonsai_species`、`help_articles` を編集可能
- **Django Admin**: `is_staff` または `role == 'admin'` のユーザーのみアクセス可能

### 2.3 権限分離

| データ | 管理者 | 一般ユーザー |
|--------|--------|--------------|
| bonsai_species（品種マスタ） | 作成・編集・削除 | 参照のみ |
| help_articles（お役立ち記事） | 作成・編集・削除・公開管理 | 参照のみ |
| article_species_relations | 編集 | 参照のみ |
| bonsai_plants, care_schedules, care_logs, bonsai_media, tags | 参照（必要時） | 自分のデータのみ編集 |

## 3. 非機能要件

| 項目 | 要件 |
|------|------|
| **セキュリティ** | HTTPS必須、Django の認証・権限でユーザー単位のデータ分離 |
| **パフォーマンス** | 一覧・詳細の初回表示 3秒以内、画像は適切なリサイズ・遅延読み込み |
| **可用性** | 採用 PaaS の SLA に依存（99.9%程度） |
| **スケーラビリティ** | 数百ユーザー規模を想定 |
| **監視** | PaaS ダッシュボード、エラーは Sentry 等を検討 |

## 4. データベース設計の技術的補足

- **ID**: UUID 推奨（`uuid.uuid4`、分散環境・マージに有利）
- **日時**: タイムゾーン考慮のため `DateTimeField` + `USE_TZ = True`
- **repeat_rule**: `JSONField` で柔軟に格納
- **全文検索**: 記事検索は `SearchVector` / `pg_trgm` または将来的に Meilisearch を検討

---

## 5. スケジュール・通知の実装方針

- **Celery Beat** で定期タスク実行（例: 5分毎に `next_run_at` をスキャン）
- 通知時間帯（notify_start_time / notify_end_time）はタスク内でフィルタ
- メール送信は **Resend** や **SendGrid** 等の SMTP 経由
- PaaS で Redis が利用可能な場合、Celery のブローカーに Redis を使用

## 7. Django Admin の活用

- **bonsai_species**: 品種マスタの一括登録・編集
- **help_articles**: 記事の作成・Markdown 編集・公開ステータス管理
- **article_species_relations**: 記事と品種の関連付け
- 管理者（`role == 'admin'`）のみ `/admin/` にアクセス可能とする

## 8. 今後の検討事項

- **Web Push**: ブラウザ通知は Service Worker + Web Push API
- **PWA**: オフライン閲覧・ホーム画面追加は将来対応可能
- **多言語**: django-modeltranslation や django-parler を検討
- **SPA 化**: 必要に応じて DRF を API として、別途 React/Vue フロントを構築
