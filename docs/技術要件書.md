## 2. 推奨技術スタック（Django・PaaS向け）

### 2.1 技術一覧

| レイヤー | 技術 | 選定理由 |
|----------|------|----------|
| **バックエンド** | Django 5.x | 成熟したフレームワーク、Admin でメタデータ管理が容易 |
| **API（Phase 2 以降）** | Django REST Framework (DRF) | **MVP では不採用**。外部アプリ／モバイル化が必要になった段階で切り出す |
| **言語** | Python 3.12+ | Django の標準 |
| **DB** | PostgreSQL（開発: Docker Compose `postgres:16` / 本番: Supabase Postgres） | トランザクション、JSONB、全文検索。dev/prod parity を維持するため開発も PostgreSQL に揃える |
| **認証** | django-allauth | メール+パスワード、Google OAuth（MVP）。Apple Sign In は Phase 2 以降 |
| **画像ストレージ** | django-storages + Supabase Storage（本番）／ ローカル `MEDIA_ROOT`（開発） | トラフィック増加時に **Cloudflare R2 + CDN** へ移行（egress 無料） |
| **非同期処理（Phase 3 以降）** | django-celery-beat + Celery / Upstash Redis（または django-q2 / Huey） | **MVP では未導入**。通知機能（Phase 3）の実装時に同時導入 |
| **UI** | Django テンプレート + Tailwind CSS（本番は CLI / PostCSS ビルド）+ HTMX + Alpine.js | サーバーサイドレンダリング、部分置換と軽量なクライアント状態（[レビュー対応案.md](./レビュー対応案.md) §4 参照）。MVP は Django テンプレート + HTMX のみで完結 |
| **デプロイ** | Render（採用）／ Supabase（DB・Storage） | `main` ブランチ push の自動デプロイ。Render 無料枠で MVP 検証、必要に応じて有料化 |
| **エラー監視** | Sentry（本番初日から導入） | 無料枠 5,000 events/月、Slack 連携で 1 人開発でも即対応可能 |
| **環境変数** | django-environ | `.env` のパース・型変換が直感的、`.env.example` をリポジトリに含める |

### 2.2 認証・権限フロー

- **django-allauth** で `email`（メール+パスワード）と `google` をサポート。**Apple Sign In は Phase 2 以降**（Web only なら必須ではないため）。
- **カスタム User モデル（AbstractUser を継承）** を `apps/accounts/models.py::User` として最初に定義。`username` フィールドは廃止し、**email をログイン ID として使用**。
- `display_name`、`time_zone`、`role`、通知設定などは User モデル本体に統合する（1:1 の `UserProfile` 分離はしない）。
- **role**: `admin` / `user`。管理者のみ `bonsai_species`、`help_articles`、`monthly_advices` を編集可能。
- **Django Admin**: `is_staff` または `role == 'admin'` のユーザーのみアクセス可能。
- **初回管理者**: `python manage.py createsuperuser` で手動作成（運用者は1〜数名規模のため自動化不要）。

### 2.3 権限分離

| データ | 管理者 | 一般ユーザー |
|--------|--------|--------------|
| bonsai_species（品種マスタ） | 作成・編集・削除 | 参照のみ |
| help_articles（お役立ち記事） | 作成・編集・削除・公開管理 | 参照のみ |
| article_species_relations | 編集 | 参照のみ |
| bonsai_plants, care_schedules, care_logs, bonsai_media, tags | 参照（必要時） | 自分のデータのみ編集 |

### 2.4 フロントエンド・インタラクション

- **HTMX**: 一覧のフィルタ・ソート・ページネーション、フォームの部分更新など、HTML フラグメントの差し替え。
- **Alpine.js**: タブ開閉・ドロップダウンなど局所的な UI 状態（複雑な SPA にはしない）。
- **Tailwind CSS**: 本番では `cdn.tailwindcss.com` を使わず、`@tailwindcss/cli` または PostCSS でビルドし、`content` にテンプレートパスを指定して未使用 CSS をパージする。

## 3. 非機能要件

| 項目 | 要件 |
|------|------|
| **セキュリティ** | HTTPS必須、Django の認証・権限でユーザー単位のデータ分離。主キーに UUIDv7 を採用し URL 列挙攻撃を防御 |
| **パフォーマンス** | 一覧・詳細の初回表示 3秒以内、画像は適切なリサイズ（長辺 2048px）・遅延読み込み（`loading="lazy"`） |

| **監視** | **Sentry を本番デプロイ初日から導入**（無料枠 5,000 events/月）。Slack 通知連携で即対応。Render ダッシュボードでメトリクスを確認 |

## 4. データベース設計の技術的補足

- **ID**: **UUIDv7 採用**（時系列ソート可能・B-tree インデックス親和性が高く URL 列挙攻撃も防げる）。`uuid7` ライブラリを利用し、`models.UUIDField(default=uuid7)` で定義。`bonsai_species` 等の共有マスタは URL 識別子として **slug を併用**。
- **日時**: タイムゾーン考慮のため `DateTimeField` + `USE_TZ = True`、`TIME_ZONE = "Asia/Tokyo"`（MVP は JST 固定）。`User.time_zone` フィールドはデフォルト `"Asia/Tokyo"` で用意し、Phase 3 通知導入時に有効化。
- **repeat_rule**: `JSONField` で柔軟に格納（iCal RFC5545 形式に準拠）。
- **全文検索**: 記事検索は `SearchVector` / `pg_trgm` または将来的に Meilisearch を検討。

### 4.1 ペルソナ起点で必要なスキーマ拡張

| 拡張項目 | 配置 | 用途 | 主要ペルソナ | フェーズ |
|---------|------|------|------------|---------|
| `bonsai_species.monthly_tasks` (JSONB) | bonsai_species | 月別の推奨作業 `[{month, period, task_type, description}]` | P1, P2 | MVP |
| `monthly_advices` 新テーブル | 新規 | 月別の共通アドバイス（month, advice_text, category） | P1 | MVP |
| **`care_schedule_completions` 新テーブル** | **新規** | **月単位の完了状態 `{user_id, source_type, source_ref, year_month, status, completed_at}`。品種マスタ由来の仮想タスク（行が存在しない）にも完了印を付けるため `care_schedules` 本体ではなく別テーブル化** | P1, P3, P5 | MVP |
| `fertilizers` 新テーブル | 新規 | 肥料マスタ（name, form_type, N, P, K, is_organic, note） | P2, P4 | MVP |
| `care_logs.fertilizer_id` (FK NULLABLE) | care_logs | 施肥ログ時の使用肥料参照 | P2, P4 | MVP |
| `bonsai_groups` / `bonsai_group_members` | 新規 | 寄せ植え対応 | P2, P5 | Phase 2 |

### 4.2 ペルソナ起点で必要なAPI／ビュー

> **注**: MVP では DRF API は採用せず、すべて Django ビュー（HTMX フラグメントレスポンスを含む）で実装する。下表のパスは将来 DRF 切り出し時の参考設計として残し、MVP では同等の URL を Django ビューで提供する。

| ビュー（MVP）／API（Phase 2 以降） | 内容 | 主要ペルソナ | フェーズ |

| `GET /reviews/last-month/` | 月末レビュー（完了率・スキップ） | P3, P5 | Phase 2 |

## 5. スケジュール・通知の実装方針

> **MVP では通知は実装しない**（[開発前検討事項.md](./開発前検討事項.md) §1.1, §5）。Celery / Redis も導入せず、本節の内容は **Phase 3** で実装する。MVP ではスケジュールの「次回予定」は都度計算（オンデマンド）し、UI 表示のみ提供する。

### 5.1 Phase 3 通知の実装方針

- **Celery Beat（または django-q2 / Huey）** で定期タスク実行（例: 5分毎に `next_run_at` をスキャン）
- ブローカー: **Upstash Redis**（マネージド、無料枠あり）
- 通知時間帯（`User.notify_start_time` / `notify_end_time`）はタスク内でフィルタ
- メール送信は **Resend** や **SendGrid** 等の SMTP 経由
- 通知導入と同時に、タイムゾーン方針を A（JST 固定）→ B（`User.time_zone` で個別保持）へ移行

### 5.2 通知駆動UX（P6 タカシ）への対応 — Phase 3

- **ディープリンク**: メール／Webプッシュの本文／payload に `https://app/todos/{todo_id}` 形式のリンクを埋め込む
- **プリフィル遷移**: ディープリンクの開封後、対象盆栽・作業種別・日時を自動入力したログ記録画面に遷移
- **ワンタップ完了**: 通知本文に「完了」アクションボタンを設置（メールは MagicLink 形式、プッシュは Action Buttons）→ アプリを開かずログ記録できる導線も検討
- **時間帯制御**: ユーザーの `notify_start_time` / `notify_end_time` を厳守（朝早すぎ・夜遅すぎが逆効果）

## 9. 画像・年間ビュー・スタイル（本番向け）

### 9.1 画像パイプライン

- **方針**: **A（同期生成）** を採用（[開発前検討事項.md](./開発前検討事項.md) §10.3 決定）。リサイズ処理は `apps/bonsai/services/images.py::generate_variants(uploaded_file)` に切り出し、後で B（Celery 非同期）/ C（オンザフライ）へ差し替え可能にする。
- **フロー**: アップロード → バリデーション（**最大 10MB**、**長辺 2048px**を超えるものはサーバー側で縮小）→ Pillow でリサイズ → **django-storages** 経由で **Supabase Storage**（本番）／ローカル `MEDIA_ROOT`（開発）へ保存。
- **生成サイズ規格（3 段）**:
  - 元画像（保管用、長辺 2048px に縮小済み）
  - 中サイズ（1080px、詳細表示用）
  - サムネ（320px、一覧用）
- **表示**: 一覧サムネイルには `loading="lazy"` を標準適用。
- **将来移行**: トラフィック増加時は **Cloudflare R2 + CDN** へ移行（egress 無料）。画像規格を頻繁に変えたい／海外配信が増えたら **C（imgproxy / Cloudflare Image Resizing）** へ移行。

### 9.2 年間スケジュール / ガント

- 月跨ぎ・動的期間が複雑になる場合は **Frappe Gantt 等のライブラリ**を評価。要件が表形式に近ければ CSS Grid ベースの拡張も継続検討。データ増加時は期間絞り込み・仮想化でレンダリング量を抑制（[レビュー対応案.md](./レビュー対応案.md) §4.4）。

## 10. 「今月のやること」生成ロジック（MVP 実装方針）

[開発前検討事項.md](./開発前検討事項.md) §10.1 の決定に基づき、**A（オンデマンド計算）+ 完了状態のみ別テーブル `care_schedule_completions` に保存** する方式を採用する。

### 10.1 合成対象

| ソース | データ取得元 | 取得方式 |
|--------|-------------|---------|
| ① 品種マスタ由来 | `bonsai_species.monthly_tasks` (JSONB) | 都度 JSONB を展開 |
| ② ユーザー個別スケジュール | `care_schedules.next_run_at` | 当月内の行を抽出 |
| ③ 月次共通アドバイス | `monthly_advices.month` | 当月の行を抽出 |

### 10.2 完了状態の管理

`care_schedule_completions`（新規テーブル）:

| カラム | 型 | 用途 |
|--------|-----|------|
| id | UUIDv7 | PK |
| user_id | FK(User) | 完了したユーザー |
| source_type | CharField | `species_task` / `schedule` / `advice` |
| source_ref | CharField | 元の参照（例: `species_task:{species_id}:{month}:{task_index}`） |
| year_month | DateField (1日固定) | 完了対象の年月 |
| status | CharField | `done` / `skipped` / `pending` |
| completed_at | DateTimeField | 完了日時 |
| log_id | FK(care_log, nullable) | プリフィル経由でログを作成した場合の参照 |

→ 品種マスタ由来の仮想タスク（DB 行が存在しない）にも完了印を付けられるよう、参照は `source_type + source_ref` の文字列キーで保持する。

### 10.3 実装サービス層

`apps/schedules/services/todos.py`:

```python
def compose_monthly_todos(user: User, year_month: date) -> list[Todo]:
    """ユーザー作成 + 品種マスタ由来 + 月次共通アドバイスの3ソースから合成。
    完了状態は care_schedule_completions と LEFT JOIN して付与。"""
```

→ 品種マスタ更新は次回リクエスト時に**即時反映**される（運営 P7 のキラー要件）。
