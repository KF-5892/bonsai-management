# 多角的レビュー — 対応案

[レビュー改善提案.md](./レビュー改善提案.md) に挙がった各項目について、**推奨対応案**と**決定が必要な候補**を整理したものです。優先度は元ドキュメントに従います。

---

## 1. UI/UXデザイン

### 1.1 モバイルファースト対応 【高】

**対応案**

- **ブレークポイント（案）**: `sm`（640px）未満を「片手スマホ」、`md`（768px）以上で一覧表・ガントの広幅レイアウトを許可。方針をドキュメントに明記する。
- **狭幅でのスケジュール**: 横長テーブルは非表示または折りたたみとし、次のいずれかを標準とする（実装時に A か B を選択）。
  - **A**: 月タブ（または月セレクト）＋その月の**縦リスト**（植物名・作業種別・要約）。
  - **B**: **植物別カード**（カード内に当月の作業チップのみ表示）。
- デスクトップ用の年間表・ガントは現行モックを維持し、**同一 URL でビュー切替**（CSS + テンプレート分岐）を想定。

**HTML モック**: [`ui-mockup.html`](./ui-mockup.html) の `#schedule-mobile` を参照。

---

### 1.2 ページ遷移・ナビゲーション設計 【高】

**対応案**

- **情報設計**: `マイ盆栽（Hub）` → `盆栽詳細` → 同一詳細内で `作業ログ` / `年間スケジュール` / `メディア` をタブまたはセカンダリナビで切替。スケジュール編集は詳細または専用サブ画面に集約。
- **パンくず**: `マイ盆栽 > {盆栽名またはID}` を詳細以上で表示。一覧へ戻るリンクをパンくず左端またはヘッダーに固定。
- **戻り導線**: ブラウザバックに加え、各サブ画面に「マイ盆栽に戻る」を明示。

**HTML モック**: `#bonsai-detail` のパンくずを参照。

---

### 1.3 盆栽詳細画面のモックアップ作成 【高】

**対応案**

- **HTML モックを追加**（一覧と同ファイル内で可）。代表画像、基本情報、体調バッジ、ログ一覧の抜粋、作業ログ入力フォームのワイヤーを含める。
- **スケジュール編集**は MVP 範囲に応じて別モック化するか、詳細内「スケジュール」タブに編集フォームの静的案を置く。

**HTML モック**: `#bonsai-detail`。

---

### 1.4 アクセシビリティ 【中】

**対応案**

- **状態表示**: 色に加え**テキストラベル**（良好／要注意／不調）と**アイコン**（例: ○／!／✕ や Material Symbols）を必ず併用。
- **コントラスト**: バッジ背景と文字色は WCAG 2.1 AA（通常テキスト 4.5:1）を満たすようデザイントークンで指定。リリース前に自動チェック（axe 等）を推奨。

---

### 1.5 空状態・オンボーディング 【中】

**対応案**

- **盆栽 0 本**: ヒーロー文言（例:「最初の盆栽を登録して、年間の手入れ予定を管理しましょう」）と主 CTA「盆栽を登録」。
- **ログ 0 件**: 「まだ記録がありません。今日の手入れを記録」＋ログ入力への誘導。
- 各リスト系画面で**データなし**用の短い説明と次アクションをテンプレート化。

**HTML モック**: `#empty-state`。

---

## 2. マーケティング・事業戦略

### 2.1 差別化ポイントの明確化 【高】

**対応案**

- **キラーフィーチャー（案の文言）**: 「品種・個体に紐づく**年間スケジュール**（表／ガント）で、潅水・施肥・手入れの時期を一目で把握」。
- **ペルソナ（3 段）**:
  - 初心者: 品種マスタ参照と通知で逃さない。
  - 中級者: ログと写真で複数本を運用。
  - 専門家: 年間ビュー・作業種別の細分化・植え替え判断のための履歴。
- **競合比較表**: 行に本アプリ・Planta・GreenSnap 等、列に「盆栽特化」「年間スケジュール」「作業ログ・写真」「品種 DB」「オフライン」などを置き、差を記載。

---

### 2.2 ソーシャル機能の将来検討 【低】

**対応案**

- **Phase 3 以降**の候補として文書に留める。MVP ではスコープ外と明記。
- **保留理由（例）**: モデレーション・プライバシー・開発コスト。需要が検証できた段階で再評価。

---

### 2.3 スケーラビリティ想定の見直し 【中】

**対応案**

- **規模の再定義**: 同時接続より**データ量（ログ・画像）**が支配的。ユーザー数は「**初期: 数百〜数千アクティブ**」を目安にし、プロダクトが広がったら読み取りレプリカ・オブジェクトストレージ・CDN を検討。
- **インフラ**: 現行の Django + PostgreSQL + PaaS で**まずは垂直スケール**で足りる範囲とし、ボトルネックが出たらキャッシュ・非同期処理を追加。

---

### 2.4 多言語対応の判断 【低】

**対応案**

- [開発前検討事項.md](./開発前検討事項.md) §7.1 で**早期に決定**。日本語のみ先行する場合は、**理由**（初期市場・リリース速度・翻訳コスト）と**将来 i18n 化の条件**（海外ユーザー比率・収益）を記載。

---

## 3. データ設計

### 3.1 care_logs の画像複数対応 【高】

**対応案**

- **推奨: 案 A** — `bonsai_media` に `log_id`（FK, NULL 可）を追加し、1 ログに複数メディアを紐付ける。既存のメディア一覧・代表画像ロジックと統合しやすい。
- **案 B**（JSON）はクエリ・制約が弱いため、複雑な検索が不要でもまずは案 A を優先。

---

### 3.2 soft delete の統一 【中】

**対応案**

- **`deleted`（users）**: アカウント無効化。**本人には非表示**、法的・監査要件があれば別途。
- **`archived`（bonsai_plants）**: 枯死・譲渡など。**一覧からは除外**しつつ、履歴参照は許可するかポリシーで決定。
- 新規テーブルでは **論理削除が必要なら** `deleted_at`（日時 NULL）に寄せるか、既存に合わせて意味を文書化する。

---

### 3.3 時系列データの成長対策 【中】

**対応案**

- **一覧**: 必ず**ページネーション**（カーソル or オフセット）と**日付範囲フィルタ**を前提にする。
- **長期**: 数年分が重くなったら**アーカイブテーブル**または**パーティション**を検討（閾値は運用で決定）。

---

### 3.4 main_image_url と bonsai_media の同期 【中】

**対応案**

- **案 1（推奨）**: アプリ層で「代表に設定したメディア」を更新したときに `main_image_url` を同期。一覧は高速。
- **案 2**: `main_image_url` を廃止し、**最新の `bonsai_media`** をクエリで取得（インデックス: `bonsai_id`, `taken_at` 等）。

---

### 3.5 検索インデックス戦略 【低】

**対応案**

- **全文検索対象の一覧**: `help_articles`（本文）、`care_logs.note`（メモ）を候補に明記。
- **方針**: PostgreSQL の `SearchVector` / `pg_trgm` で開始し、負荷が高ければ Meilisearch 等を検討。対象カラムとインデックスを設計書に一覧化。

---

## 4. フロントエンド実装

### 4.1 インタラクション方針の決定 【高】

**対応案**

- **HTMX**: 部分置換（フィルタ・ソート・ページネーション・モーダル読み込み）。
- **Alpine.js**: タブ開閉・軽量なクライアント状態。複雑な SPA にはしない。

---

### 4.2 Tailwind CSS のビルド方針 【高】

**対応案**

- 本番は **`@tailwindcss/cli` または PostCSS** でビルドし、`content` にテンプレートパスを指定して**未使用 CSS をパージ**。
- 開発中のみ CDN を許容する運用でも可だが、本番バンドルに CDN は使わない。

---

### 4.3 画像処理パイプライン 【高】

**対応案**

- **フロー**: アップロード → バリデーション → **リサイズ・サムネイル生成**（ImageKit / `django-imagekit` / 自前 Pillow）→ **S3 互換** → **CDN** 配信。
- **表示**: `loading="lazy"` を一覧のサムネイルに標準適用。

---

### 4.4 ガントチャートの実装方針 【中】

**対応案**

- **判断基準**: 月跨ぎ・動的期間が複雑なら **Frappe Gantt 等のライブラリ**を評価。要件が表形式に近ければ **CSS Grid の拡張**も継続検討。
- データ増加時は**仮想スクロール**や**期間の絞り込み**でレンダリング量を制御。

---

### 4.5 オフライン対応の優先度 【低】

**対応案**

- **PWA / Service Worker** は **Phase 2 以降**。MVP ではオフライン記録は行わず、ネットワーク不安定時はエラーメッセージと再送方針のみ検討。

---

## 5. 盆栽専門家のニーズ

### 5.1 作業種別の拡充 【高】

**対応案**

- **初期 Choices に追加する候補**: 芽摘み、葉刈り、葉水、針金かけ、針金外し、消毒、ジン・シャリ作り、取り木、挿し木（既存の潅水・施肥・剪定・植え替え・観察に加える）。
- **カスタム種別**: Phase 2 で `user_id` 付きカスタム task テーブル、または自由入力＋正規化を検討。MVP は拡張 Choices のみでも可。

---

### 5.2 天候・気温との連動 【高】

**対応案**

- **最低限**: `care_logs` に **`weather_note`**（文字列）と **`temperature_c`**（数値 NULL 可）を追加し、手入力で記録可能にする。
- **将来**: OpenWeatherMap 等で**当日の参照値を提案**する API 連携は別機能として設計。

---

### 5.3 写真の時系列比較 【中】

**対応案**

- **UI**: 同一画面で**ビフォーアフター（スライダー）**または**並列表示**。**前年同月**の写真を出すビューは `bonsai_media.taken_at` を利用。
- MVP 後でも可。データは `bonsai_media` に既にある前提。

---

### 5.4 植え替え専用サマリー 【中】

**対応案**

- **専用ビューまたは詳細内パネル**: 最終植え替え日、次回目安、用土メモ（`soil_info` またはログから抽出）。
- **アラート**: 次回予定が近い盆栽を一覧で強調。

---

### 5.5 展示会管理 【低】

**対応案**

- **MVP 後・将来機能**。カレンダー連携や出品リストは要求が固まってからスキーマを設計。
