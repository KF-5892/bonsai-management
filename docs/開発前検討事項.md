## 3. 認証・ユーザーモデル

### 3.1 User モデル設計

| 検討項目 | 選択肢 | 決定 |
|----------|--------|------|
| モデル構成 | Django 標準 User + UserProfile（1:1）<br>カスタム AbstractUser で統合 | |
| ログインID | email をログインIDとして使用（username は廃止 or 自動生成） | |

### 3.2 UserProfile の項目

データ定義の `users` テーブルとの対応:

| データ定義 | Django での保持先 |
|------------|-------------------|
| display_name | UserProfile または User |
| time_zone, language | UserProfile |
| notification_enabled, notify_start_time, notify_end_time | UserProfile |
| role | UserProfile または User（is_staff と連動） |
| auth_provider, provider_user_id | django-allauth の SocialAccount で管理 |

### 3.3 OAuth 準備

| プロバイダ | 必要な作業 | 状態 |
|------------|------------|------|
| Google | Google Cloud Console で OAuth 2.0 クライアント作成、リダイレクト URI 登録 | |
| Apple | Apple Developer アカウント、Sign in with Apple 設定 | |

### 3.4 初回管理者の作成

| 検討項目 | 選択肢 | 決定 |
|----------|--------|------|
| 方法 | `createsuperuser` で手動作成<br>マイグレーション or 管理コマンドで seed | |

## 4. データ・モデル

### 4.1 主キー

| 検討項目 | 選択肢 | 決定 |
|----------|--------|------|
| 形式 | UUID（技術要件書推奨）<br>整数（Django デフォルト） | |

### 4.2 品種マスタの初期データ

| 検討項目 | 選択肢 | 決定 |
|----------|--------|------|
| 方針 | 代表的な品種（黒松、真柏など）を fixture / 管理コマンドで投入<br>運用開始後に Admin で手動登録 | |

### 4.3 種別の定義

| 項目 | 選択肢 | 決定 |
|------|--------|------|
| task_type | Choices で固定（潅水、施肥、剪定、植え替え、観察）<br>自由入力 | |
| task_type 拡張候補（レビュー） | 芽摘み、葉刈り、葉水、針金かけ、針金外し、消毒、ジン・シャリ作り、取り木、挿し木 等を初期 Choices に含めるか | 詳細は [レビュー対応案.md](./レビュー対応案.md) §5.1 |
| repeat_type | Choices で固定（none, daily, weekly, monthly, yearly, custom） | |
| category（品種） | Choices で固定（針葉樹、広葉樹、花物、実物）<br>自由入力 | |
| health_status | Choices で固定（良好、要注意、不調）<br>自由入力 | |

## 7. その他

### 7.1 言語

| 検討項目 | 選択肢 | 決定 |
|----------|--------|------|
| 初回 | 日本語のみ / 最初から i18n 対応 | |

**日本語のみ先行する場合の記載テンプレ**（決定欄または別紙に記載）

- **初期ターゲット市場**: （例: 国内ユーザー中心）
- **i18n を後回しにする理由**: リリース速度、翻訳・運用コスト、検証フェーズの範囲
