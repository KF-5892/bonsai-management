## 10. 設計論点の整理（実装前に判断が必要な3項目）

実装に入る前に、以下3項目について方針を決定する。各項目の論点・選択肢・判断基準を以下に整理する。

### 10.1 「今月のやること」計算ロジック

#### 10.1.1 何が論点か

「今月のやること」は次の3つのソースから合成される。**どのレイヤーで／どのタイミングで合成するか**を決める必要がある。

| ソース | 内容 | 例 |
|--------|------|-----|
| ① 品種マスタ由来 | `bonsai_species.monthly_tasks` (JSONB) の月別推奨作業 | 黒松・5月：芽摘み |
| ② ユーザー個別スケジュール | `care_schedules` のユーザーが登録した繰り返しタスク | 「私の黒松A」に毎週月曜潅水 |
| ③ 月次共通アドバイス | `monthly_advices` の品種に依らない月別アドバイス | 5月：水切れ注意 |

#### 10.1.2 選択肢（合成方式）

| 案 | 合成のタイミング | 保存方式 | 長所 | 短所 |
|----|--------------|---------|------|------|
| **A. オンデマンド計算** | API/ビュー呼び出し時に毎回計算 | 都度生成、保存しない | 実装シンプル、常に最新の品種マスタを反映 | 毎回 JOIN/JSONB 展開のコスト、完了状態の保持に別テーブルが必要 |
| **B. 月初バッチで事前展開** | Celery Beat で月初に全ユーザー分を `care_schedules` に展開 | 通常の行として保存 | 完了/スキップを通常のレコードで管理しやすい、参照が高速 | 品種マスタ更新の反映に再展開が必要、Celery 必須、データ量が膨らむ |
| **C. ハイブリッド** | ①は都度計算（マスタ参照）／②③は通常テーブル／完了状態は別テーブル `care_schedule_completions` | 完了状態だけ保存 | 各ソースの性質に合った扱い、マスタ更新が即反映 | レイヤー設計がやや複雑（サービス層の責務が増える） |

#### 10.1.3 判断基準

| 観点 | A が向く場合 | B が向く場合 | C が向く場合 |
|------|-------------|-------------|-------------|
| **MVP のスコープ** | Celery を導入したくない、最小構成で出したい | すでに通知で Celery を使う前提 | 中長期的に拡張する前提 |
| **品種マスタの更新頻度** | 頻繁に更新したい（即反映が重要） | ほぼ更新しない | 頻繁に更新したい |
| **完了状態の管理** | テーブル設計を増やしたくない | 1テーブル（`care_schedules`）に集約したい | 完了履歴を別軸で集計したい |
| **データ量** | アクティブユーザーが少ない初期 | ユーザー数が多くなっても参照を高速にしたい | バランス重視 |

#### 10.1.4 推奨（仮）

**MVP では C（ハイブリッド）を推奨**。
理由:
- 品種マスタ由来のタスクは P1（入門者）にとってのキラー機能であり、運営（P7）が更新したら**即時反映**したい → 都度計算が望ましい。
- ユーザーの個別スケジュールは普通の RDB 行として持ち、繰り返しは `repeat_rule` (JSONB) で表現すれば既存の Django パターンで足りる。
- 完了状態を別テーブル（`care_schedule_completions`：`{user_id, source_type, source_ref, year_month, status, completed_at}`）にすると、品種マスタ由来のタスク（行が存在しない）にも完了マークを付けられる。

ただし、Celery 導入を MVP で見送る場合は **A（オンデマンドのみ）** で開始し、必要が出てから C に拡張する道も合理的。

#### 10.1.5 決定欄

| 検討項目 | 選択肢 | 決定 |
|----------|--------|------|
| 合成方式 | A（オンデマンド）／B（月初バッチ）／C（ハイブリッド） | **A（オンデマンド計算）+ 完了状態のみ別テーブル**（実質 C のシンプル版）。MVP 規模なら毎リクエスト計算でも体感問題なし。Celery 不採用方針とも整合 |
| 完了状態の保持 | `care_schedules` 行に保持 / 別テーブル `care_schedule_completions` | **別テーブル `care_schedule_completions`**（品種マスタ由来の仮想タスクにも完了印を付けるため。`{user_id, source_type, source_ref, year_month, status, completed_at}`） |
| 品種マスタ更新の反映 | 即時（都度計算）/ バッチ（再展開時） | **即時（都度計算）**（運営 P7 が更新したら即反映が必要） |

### 10.2 タイムゾーン方針

#### 10.2.1 何が論点か

盆栽管理アプリでは「**今日のやること**」「**月次推奨作業**」「**通知時間帯（朝7時〜夜9時など）**」がいずれも**ユーザーのローカル時刻**に依存する。サーバー側で時刻をどう持ち、どう変換するかの方針が必要。

#### 10.2.2 Django の基本前提

- `USE_TZ = True` が Django 5.x の標準。
- DB には**常に UTC で保存**される（PostgreSQL の `TIMESTAMPTZ`）。
- 表示・通知時にユーザーのタイムゾーンへ変換する。

#### 10.2.3 選択肢

| 案 | ユーザー TZ の保持 | 適用範囲 | 長所 | 短所 |
|----|------------------|---------|------|------|
| **A. 単一 TZ（Asia/Tokyo 固定）** | 保持しない | 全ユーザー JST 固定 | 実装最小、設定不要 | 国内専用、海外移住・出張ユーザー非対応、将来 i18n 化で改修必要 |
| **B. UserProfile.time_zone** | DB に文字列で保存（例 `"Asia/Tokyo"`） | リクエスト毎に middleware で `activate()` | 標準的、将来海外対応もそのまま使える | ユーザー設定 UI が必要、登録時のデフォルト判定（IP / ブラウザ）を考えるべき |
| **C. ブラウザ TZ を都度送信** | 保存しない／JS で都度送信 | フロント側で表示変換、サーバーはほぼ UTC のまま | サーバーがシンプル | 通知（バックエンド単独動作）でユーザー TZ がわからない → 通知時間帯制御ができない |

#### 10.2.4 判断基準

| 質問 | Yes なら… |
|------|----------|
| 通知（朝7時〜夜9時など時間帯制御）を MVP で出すか？ | **A または B**（C はサーバー単独で TZ を知れないため不適） |
| 国内ユーザー専用で当面進めるか？ | **A** で十分（将来必要になったら B へ移行） |
| 海外ユーザー、または海外旅行・出張中の通知を考慮するか？ | **B** が必要 |
| Phase 1 で通知を含めず、ユーザー設定 UI を作る余裕がないか？ | **A** で開始、Phase 2 で B へ移行 |

#### 10.2.5 推奨（仮）

**MVP では A（Asia/Tokyo 固定）を推奨**。
理由:
- MVP 対象ペルソナ（P1/P2/P7）はいずれも国内ユーザー前提（`docs/ユーザーペルソナ.md` 参照）。
- 通知が MVP に含まれない場合、TZ 切替の必要性は実質ゼロ。
- `settings.py` で `TIME_ZONE = "Asia/Tokyo"`、`USE_TZ = True`、UserProfile に `time_zone` フィールドだけ用意して**デフォルト値 `"Asia/Tokyo"` で固定**（後で B に移行する余地を残す）。

ただし、通知を MVP に含めるなら最初から **B** を選択し、UserProfile に `time_zone` を実際に使う形で実装すべき。

#### 10.2.6 決定欄

| 検討項目 | 選択肢 | 決定 |
|----------|--------|------|
| TZ 方針 | A（JST 固定）/ B（UserProfile で保持）/ C（ブラウザ送信） | **A（JST 固定）で開始**。Phase 3 の通知導入時に B 方式へ移行 |
| `USE_TZ` | True（推奨）/ False | **True**（Django 5.x 標準、DB は UTC で保存） |
| `TIME_ZONE` 設定値 | `"Asia/Tokyo"` / `"UTC"` | **`"Asia/Tokyo"`** |
| UserProfile.time_zone | フィールドを用意するか（用意するなら A でもデフォルト固定） | **フィールドは用意する**（`CharField(default="Asia/Tokyo")`）。Phase 3 の B 方式移行は migration 1 本で済むようにしておく |

### 10.3 画像のリサイズ・サムネイル生成のタイミング

#### 10.3.1 何が論点か

ユーザーがスマホで撮影した盆栽の写真は数 MB〜十数 MB に及ぶ。一覧表示用のサムネイルや詳細表示用の中サイズ画像をどう／いつ生成するか。

#### 10.3.2 選択肢

| 案 | 生成タイミング | 実装方式 | 長所 | 短所 |
|----|-------------|---------|------|------|
| **A. アップロード時に同期生成** | リクエスト処理中に Pillow でリサイズ | `save()` メソッドや signal で実行 | 実装シンプル、Celery 不要、保存後すぐ全サイズ揃う | アップロードレスポンスが遅延（数百ms〜数秒）、巨大画像で worker タイムアウトのリスク |
| **B. アップロード後に非同期生成** | アップロード完了後 Celery で生成 | `django-imagekit` のキャッシュバックエンドを Celery 化 / カスタム task | レスポンス即時、巨大画像も安全、リトライ可能 | Celery 必須、生成完了までサムネ未表示の窓ができる（プレースホルダ要） |
| **C. 表示時に動的生成（オンザフライ）** | 初回アクセス時に CDN/サーバーで生成 | imgproxy / `sorl-thumbnail` / Cloudflare Image Resizing | 必要な解像度だけ生成、運用後の規格変更が容易 | 初回表示が遅い、外部サービス費用、自前運用ならインフラ複雑化 |

#### 10.3.3 判断基準

| 質問 | Yes なら… |
|------|----------|
| Celery を MVP に導入するか？ | **B** が選択可能 |
| Celery 不採用で、運用初期はユーザー数が少ないか？ | **A** で十分 |
| サムネイル規格（サイズ・トリミング）が変わる可能性が高いか？ | **C** が柔軟（規格変更で再生成不要） |
| 画像数が多く、ストレージ容量を抑えたいか？ | **C**（必要な解像度だけ） |
| 「アップロード即一覧反映」のシビアな UX 要求があるか？ | **B**（プレースホルダで埋める）または **C** |
| 外部画像処理サービスの費用を払いたくないか？ | **A** または **B** |

#### 10.3.4 推奨（仮）

**MVP では A（同期生成、ただしサイズ上限と Pillow 最適化を徹底）を推奨**。

具体策:
- アップロード時にバリデーション: 最大 10MB、長辺 4096px までを許可、超えたものはサーバー側で縮小して保存。
- 生成サイズ: 元画像（保管用）／中サイズ（1080px、詳細表示用）／サムネ（320px、一覧用）の3段。
- Pillow の `Image.thumbnail()` で生成し、`django-storages` 経由で保存。
- 一覧では `loading="lazy"` を必須化。

理由:
- MVP 規模（数百〜数千アクティブ）では同期処理でも体感問題が出にくい。
- Celery 導入を通知機能に合わせて判断したい場合、画像のためだけに Celery を入れるのは過剰。
- 後から **B** に移行する場合も、リサイズ処理を `services/images.py` のような関数に切り出しておけば差し替え容易。
- 画像規格を頻繁に変えたい／海外配信が増えたら **C**（imgproxy / Cloudflare Images）に移行する。

#### 10.3.5 決定欄

| 検討項目 | 選択肢 | 決定 |
|----------|--------|------|
| 生成タイミング | A（同期）/ B（Celery 非同期）/ C（オンザフライ） | **A（同期生成）**。リサイズ処理は `services/images.py::generate_variants()` に切り出し、後で B/C へ差し替え可能にする |
| ライブラリ | Pillow 直接 / django-imagekit / sorl-thumbnail / 外部サービス | **Pillow 直接 + django-storages**（Supabase Storage 接続） |
| 生成サイズ規格 | 元 + 中（1080px）+ サムネ（320px）/ その他 | **元（保管）/ 中 1080px / サムネ 320px の 3 段** |
| アップロード上限 | 10MB / 20MB / その他 | **10MB** |
| 長辺リサイズ閾値 | 4096px / 2048px / その他 | **2048px**（Instagram 実質最大値と同等、4096px は過剰） |
